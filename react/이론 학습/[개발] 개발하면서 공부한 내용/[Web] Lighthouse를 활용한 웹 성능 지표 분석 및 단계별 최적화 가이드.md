## 1. Lighthouse란?
Lighthouse는 Google에서 제공하는 자동화된 웹 품질 분석 도구다. 5가지 핵심 항목을 수치화하여 객관적인 리포트를 제공한다.
* **Performance:** 로딩 속도 및 실행 성능
* **Accessibility:** 웹 접근성 준수 여부
* **Best Practices:** 보안 및 현대적 웹 표준 준수
* **SEO:** 검색 엔진 최적화 수준
* **PWA:** 프로그레시브 웹 앱 지원 여부

## 2. Core Web Vitals: 사용자 경험의 핵심 지표
프론트엔드 개발자가 가장 집중해야 할 3대 지표다.
* **LCP (Largest Contentful Paint):** 가장 큰 요소가 뜨는 시간. **(2.5s 이내 권장)**
* **CLS (Cumulative Layout Shift):** 화면 밀림 현상 정도. **(0.1 이내 권장)**
* **INP (Interaction to Next Paint):** 사용자 클릭/입력 반응 속도. **(200ms 이내 권장)**

## 3. 왜 프로젝트에서 성능 지표를 알아야 하는가?
* **객관적 증명:** "빠르다"는 주관을 "LCP 1.5s"라는 수치로 증명할 수 있다.
* **비즈니스 가치:** 성능이 개선되면 이탈률이 줄고, 구글 검색 순위(SEO)가 올라간다.
* **기술적 역량:** 렌더링 원리를 이해하고 최적화할 줄 아는 개발자임을 보여준다.

## 4. 프로젝트 단계별 Lighthouse 활용 프로세스
나중에 프로젝트를 시작하면 이 순서대로 진행하자.

1.  **Baseline 설정 (초기):** 메인 페이지 구조가 잡히면 **시크릿 모드**에서 측정 후 기록한다.
2.  **기능 구현 중:** 이미지 파일이 커지거나 서드파티 라이브러리(지도, 차트 등)를 추가할 때 수시로 체크한다.
3.  **최적화 단계:** Lighthouse 리포트의 `Opportunities` 섹션을 보고 하나씩 해결한다.
4.  **최종 검증:** 배포 후 실제 환경(Production) 점수를 확인하고 개선 전/후를 비교한다.

## 5. 실전 성능 개선 치트 시트 (Problem-Solution)
나중에 성능 문제가 발생하면 이 표를 보고 해결책을 찾자.

| 구분 | 문제 상황 (Problem) | 해결 방법 (Technical Solution) |
| :--- | :--- | :--- |
| **LCP** | 메인 이미지가 늦게 뜸 | `next/image` 활용, 배너 이미지에 `priority` 속성 추가 |
| **LCP** | 폰트 로딩 시 글자가 늦게 나타남 | `font-display: swap` 설정 및 최적화된 웹폰트(woff2) 사용 |
| **CLS** | 이미지 로딩 후 텍스트가 아래로 밀림 | 이미지 태그에 `width`, `height` 명시 혹은 `aspect-ratio` 사용 |
| **CLS** | 동적 리스트가 뒤늦게 나타남 | **Skeleton UI**를 적용하여 미리 공간 확보 |
| **INP** | 버튼 클릭 후 반응이 한 박자 늦음 | 무거운 로직에 `useTransition` 적용 혹은 `Debounce` 활용 |
| **ETC** | 번들 파일(JS) 크기가 너무 큼 | `Dynamic Import`를 이용한 코드 스플리팅 적용 |

## 6. 성능 측정 시 주의사항 (정확한 데이터 얻기)
정확한 측정을 위해 다음 환경을 반드시 지키자.
* **시크릿 모드 사용:** 브라우저 확장 프로그램이 성능 점수를 깎아먹는 것을 방지한다.
* **일관된 환경:** 네트워크 속도 제한(Throttling) 설정을 항상 동일하게 유지한다. (보통 Mobile 환경 기준 추천)
* **Production 빌드:** 개발 모드(`npm run dev`)는 최적화가 안 되어 점수가 낮으므로, 실제 빌드본(`npm run build`)으로 측정한다.
